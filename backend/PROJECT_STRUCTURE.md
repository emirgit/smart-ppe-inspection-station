# Turnstile Backend — Project Structure

> A guide to help you understand how the codebase is organized and where things belong.

## Root Directory

```
turnstile-backend/
├── prisma/                  # Database schema & migrations
├── src/                     # Application source code
├── .env                     # Environment variables (git-ignored)
├── .gitignore
├── package.json
├── prisma.config.ts         # Prisma engine & datasource config
├── PRISMA_GUIDE.md          # Prisma usage guide for the team
└── PROJECT_STRUCTURE.md     # This file
```

---

## `prisma/` — Database Layer

```
prisma/
├── schema.prisma            # Single source of truth for all DB models
└── migrations/              # Auto-generated migration history
```

- **`schema.prisma`** — Defines all database tables, columns, types, relations, and enums. When you need to add/change a table or column, **this is the only file you edit**.
- **`migrations/`** — Stores migration SQL files generated by `prisma migrate`. These are version-controlled so every developer and deployment environment applies the same database changes in order.

---

## `src/` — Application Code

```
src/
├── config/                  # App-wide configuration & service clients
├── controllers/             # Request handlers (business logic)
├── middlewares/              # Express middleware functions
├── models/                  # Domain-specific utilities & helpers
├── routes/                  # Route definitions & endpoint mapping
├── generated/               # Auto-generated code (git-ignored)
│   └── prisma/              # Generated Prisma Client
└── server.js                # App entry point
```

### `server.js` — Entry Point

Sets up the Express app: loads environment variables, registers middlewares (helmet, cors, JSON parsing), mounts route handlers, and starts listening on the configured port.

### `config/` — Configuration

Centralized setup for external services and shared config.

| File | Purpose |
|---|---|
| `prisma.js` | Exports a **singleton Prisma Client** instance. Always import from here — never create `new PrismaClient()` directly. |

Add future config files here, e.g. `redis.js`, `firebase.js`, etc.

### `controllers/` — Business Logic

Each controller file contains handler functions for a specific resource. Controllers receive the request, interact with the database (via Prisma), and send responses.

**Naming convention:** `<resource>.controller.js`

| File | Handles |
|---|---|
| `health.controller.js` | `GET /api/health` — returns server status, uptime |

**Example pattern:**
```js
const prisma = require('../config/prisma');

const getAllWorkers = async (req, res, next) => {
  try {
    const workers = await prisma.worker.findMany({ include: { role: true } });
    res.json(workers);
  } catch (err) {
    next(err);
  }
};

module.exports = { getAllWorkers };
```

### `routes/` — Route Definitions

Each route file maps HTTP methods + paths to controller functions. Routes are mounted in `server.js` with a base path prefix.

**Naming convention:** `<resource>.routes.js`

| File | Base Path | Endpoints |
|---|---|---|
| `health.routes.js` | `/api/health` | `GET /` |

**Example pattern:**
```js
const express = require('express');
const router = express.Router();
const { getAllWorkers } = require('../controllers/worker.controller');

router.get('/', getAllWorkers);

module.exports = router;
```

### `middlewares/` — Express Middleware

Functions that run before or after route handlers. Used for cross-cutting concerns like error handling, authentication, validation, and logging.

| File | Purpose |
|---|---|
| `errorHandler.js` | Global error handler — catches errors from `next(err)`, returns JSON error responses. Hides stack traces in production. |

Add future middleware here, e.g. `auth.js`, `validate.js`, etc.

### `models/` — Domain Utilities

Reserved for domain-specific helper functions, data transformations, or business rule logic that doesn't belong in controllers. Since Prisma handles the data model, this folder is for things like:

- Custom validation logic
- Data formatting/transformation helpers
- Business rule calculations

### `generated/` — Auto-Generated Code (Git-Ignored)

Contains the Prisma Client generated from `schema.prisma`. **Never edit files here** — they are overwritten every time you run `npx prisma generate`.

## Request Flow

```
Client Request
    │
    ▼
server.js (middlewares: helmet → cors → json parser)
    │
    ▼
routes/<resource>.routes.js (maps URL → controller function)
    │
    ▼
controllers/<resource>.controller.js (business logic + DB queries via Prisma)
    │
    ▼
config/prisma.js → Prisma Client → PostgreSQL Database
    │
    ▼
Response sent back to client
    │
    ▼
middlewares/errorHandler.js (catches any errors thrown with next(err))
```

## Adding a New Feature (Checklist)

When adding a new resource (e.g. `workers`):

1. **Schema** — If new tables/columns are needed, update `prisma/schema.prisma`, run `npx prisma db push`, then `npx prisma generate`
2. **Controller** — Create `src/controllers/worker.controller.js` with handler functions
3. **Routes** — Create `src/routes/worker.routes.js` mapping endpoints to handlers
4. **Mount** — Register the route in `server.js`: `app.use('/api/workers', workerRoutes)`
5. **Middleware** — Add any resource-specific middleware to `src/middlewares/` if needed
